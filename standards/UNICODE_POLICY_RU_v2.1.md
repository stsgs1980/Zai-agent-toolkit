# Стандарт: Политика No-Unicode v2.1

> ID: STD-DOC-003
> Версия: 2.1.3
> Уровень: **[C] Критический** + **[W] Предупреждение** + **[I] Информация**
> Связан: Markdown Standard (STD-DOC-001, STD-DOC-002)

Стандарт использования символов, иконок и графики. Уровень Design System / Engineering Governance.

---

## 1. Назначение

Этот стандарт устанавливает правила использования Unicode-графики на всех слоях продукта: интерфейс, контент, код, системные коммуникации.

### Цели:

- обеспечить визуальную согласованность
- поддерживать профессиональный уровень продукта
- гарантировать управляемость через дизайн-систему
- исключить неконтролируемые визуальные артефакты

## 3. Уровни строгости

Стандарт применяет **многоуровневый подход** вместо абсолютного запрета:

| Уровень | Обозначение | Контекст | Действие |
|---------|-------------|----------|----------|
| Критический | [C] | UI, продакшн-код | Блокирует merge |
| Предупреждение | [W] | Документация, README, AI-чат | Предупреждение при review (см. MARKDOWN_STANDARD) |
| Информация | [I] | Внутренние заметки, прототипы | Рекомендация |

### Применение уровней:

| Контекст | Уровень | Обоснование |
|----------|---------|-------------|
| UI-компоненты | [C] Критический | Прямое влияние на пользователя |
| API-ответы, ошибки | [C] Критический | Отображаются в интерфейсе |
| Продакшн-код | [C] Критический | Может попасть в UI |
| Проектная документация | [W] Предупреждение | См. MARKDOWN_STANDARD |
| README-файлы | [W] Предупреждение | См. MARKDOWN_STANDARD |
| Внутренние заметки | [I] Информация | Только для разработчиков |
| Прототипы / MVP | [I] Информация | Временный код |
| Логи, отладка | [I] Информация | Не видны пользователю |
| AI-коммуникация (чат) | [W] Предупреждение | Профессиональный стиль общения агента |

**Примечание для AI-коммуникации:** Ответы AI-агента в чате не должны содержать emoji и Unicode-графику. Это обеспечивает согласованный профессиональный стиль наравне с кодом и документацией. Сообщения пользователя не регулируются этим стандартом.

## 5. Причины ограничений

| Проблема | Описание |
|----------|----------|
| Несогласованность отображения | Unicode отображается по-разному на разных ОС, браузерах, устройствах |
| Отсутствие контроля | Нельзя централизованно менять стиль, управлять темами |
| Нет масштабируемости | Нет контроля размера, нет адаптивности |
| Нарушение профессионального стандарта | Снижает доверие, нарушает визуальную иерархию |

## 6. Разрешённые символы

### 6.1. Базовый набор

| Категория | Диапазон | Примеры |
|-----------|----------|---------|
| ASCII-буквы | a-z, A-Z | text, TEXT |
| Кириллица | а-я, А-Я | текст, ТЕКСТ |
| Цифры | 0-9 | 123 |
| Пунктуация | . , ; : ! ? - _ | стандартные |
| Пробелы | пробел, таб, перевод строки | форматирование |

### 6.2. Whitelist для диаграмм (только код [I])

Для технических диаграмм в коде разрешены:

| Символ | Использование |
|--------|---------------|
| -> | стрелка вправо |
| <- | стрелка влево |
| => | импликация |
| <= | обратная импликация |
| \| | вертикальная линия |
| + | соединение линий |
| - | горизонтальная линия |
| v | стрелка вниз |
| ^ | стрелка вверх |
| > | указатель |
| < | обратный указатель |

### 6.3. Форматирование кода в комментариях и документации

При форматировании кода в комментариях и встроенной документации:

| Элемент | Формат |
|---------|--------|
| Инлайн-код | `` `code` `` |
| Блок кода | `` ```language `` |

**Правила:**

- Указывать язык для блоков кода (подсветка синтаксиса)
- Не использовать HTML-теги для окрашивания кода
- Цвет — ответственность IDE/рендерера

### 6.4. Пример разрешённой диаграммы (уровень [I])

```text
+-------------------+
|    Компонент A    |
+---------+---------+
          |
          v
+-------------------+
|    Компонент B    |
+---------+---------+
          |
          +-----> Компонент C
```

## 7. Стандарт иконок

### 7.1. Базовое правило

Любой визуальный символ в UI = **только SVG**

### 7.2. Требования к SVG

- быть частью Design System
- использовать design tokens
- поддерживать темизацию
- быть оптимизированным (SVGO)
- иметь единый стиль stroke/fill

### 7.3. Библиотеки иконок

| Библиотека | Статус | Использование |
|------------|--------|---------------|
| Lucide | Основная | Все иконки проекта |
| Брендовые логотипы | Обязательно | Технологии, интеграции |

### 7.4. Брендовые логотипы

При упоминании технологий использовать официальные SVG:

| Технология | Требование |
|------------|------------|
| Next.js | Официальный SVG-логотип |
| TypeScript | Официальный SVG-логотип |
| Tailwind CSS | Официальный SVG-логотип |
| Prisma ORM | Официальный SVG-логотип |

## 8. AI-промпты

### 8.1. Правильная формулировка

```text
Вывод должен содержать только:
- ASCII-символы (a-z, A-Z, 0-9, стандартная пунктуация)
- Кириллические символы (а-я, А-Я)
- Символы из whitelist для диаграмм (для уровня [I]): -> <- => <= \| + - v ^ > <
```

### 8.2. Очистка документов перед анализом

```javascript
// Удаление emoji и Unicode-графики
text.replace(/[\u{1F000}-\u{1FFFF}]|[\u{2600}-\u{27BF}]|[\u{FE00}-\u{FEFF}]|[\u{1F900}-\u{1F9FF}]|[\u{2702}-\u{27B0}]/gu, '')
```

### 8.3. Финальная санитизация

```javascript
// Для уровня [C] (код/UI) - только ASCII + кириллица (типографика строго запрещена)
text.replace(/[^\x20-\x7E\u0400-\u04FF]/g, '')

// Для уровня [I] - с whitelist для диаграмм
// Примечание: 'v' удалён из явного перечисления, так как он в базовом диапазоне ASCII \x20-\x7E
text.replace(/[^\x20-\x7E\u0400-\u04FF\-\>\<\=\|\+\^]/g, '')

// ВНИМАНИЕ: Уровень [W] (документация) регулируется MARKDOWN_STANDARD v2.1.
// Там типографические символы (тире, градус, копирайт) РАЗРЕШЕНЫ в обычном тексте, поэтому эта
// строгая санитизация НЕ применяется к .md файлам.
```

## 9. Стратегия Fallback

### 9.1. Когда SVG недоступен

| Ситуация | Решение |
|----------|---------|
| Ошибка загрузки SVG | Текстовая альтернатива (скрытая, aria-label) |
| Email-клиенты | Текст + стилизованный span |
| Терминал / CLI | Текст + ANSI-цвета |
| Обычный текст | Только текст |

### 9.2. Реализация Fallback

```html
<!-- SVG с fallback через onerror -->
<span class="icon">
  <svg onerror="this.style.display='none';this.nextElementSibling.style.display='inline'"
       aria-hidden="true">
    <!-- содержимое иконки -->
  </svg>
  <span class="icon-fallback" style="display:none">Сохранить</span>
</span>
```

```css
.icon-fallback {
  display: none;
}
/* Fallback показывается через JS onerror на SVG-элементе */
```

## 10. Контроль и обеспечение

### 10.1. Линтинг

Файл: `eslint-rules/no-unicode-policy.js`

```javascript
module.exports = {
  meta: {
    type: 'problem',
    docs: { description: 'Обеспечение No-Unicode Policy' }
  },
  create(context) {
    const emojiPattern = /[\u{1F000}-\u{1FFFF}]/u;
    const severity = {
      '[C]': 'error',  // UI, продакшн-код
      '[I]': 'off'     // Внутреннее, прототипы
    };

    return {
      Literal(node) {
        if (emojiPattern.test(node.value)) {
          context.report({
            node,
            message: 'Unicode-графика запрещена. Используйте SVG.'
          });
        }
      },
      TemplateLiteral(node) {
        for (const quasi of node.quasis) {
          if (emojiPattern.test(quasi.value.cooked)) {
            context.report({
              node,
              message: 'Unicode-графика запрещена в шаблонных литералах. Используйте SVG.'
            });
            break;
          }
        }
      },
      JSXText(node) {
        if (emojiPattern.test(node.value)) {
          context.report({
            node,
            message: 'Unicode-графика запрещена в JSX. Используйте SVG.'
          });
        }
      }
    };
  }
};
```

## 11. Исключения

### 11.1. Безусловно разрешены

| Категория | Примеры |
|-----------|---------|
| Буквы | a-z, A-Z, а-я, А-Я |
| Цифры | 0-9 |
| Пунктуация | . , ; : ! ? - _ ( ) [ ] { } |
| Символы whitelist [I] | -> <- => <= \| + - v ^ > < |

### 11.2. Исключения по согласованию

| Ситуация | Требование |
|----------|------------|
| Внешние требования | Email-рассылки с emoji — согласовать с маркетингом |
| Локализация | Языки с не-ASCII символами (китайский, арабский) |
| Доступность | Unicode-символы для скринридеров |

### 11.3. Процесс согласования

1. Создать issue с обоснованием
2. Получить одобрение Tech Lead
3. Задокументировать исключение в коде
4. Добавить в whitelist при необходимости

## 13. Чек-лист соответствия

### Перед merge (код [C]):

- [ ] Нет emoji в UI-компонентах
- [ ] Нет Unicode-иконок в продакшн-коде
- [ ] Иконки реализованы через SVG
- [ ] Брендовые логотипы — официальные SVG
- [ ] Fallback существует для критических иконок
- [ ] AI-промпты используют правильную формулировку

### Для документации [W]:

- [ ] См. MARKDOWN_STANDARD v2.1

### Для AI-коммуникации (чат) [W]:

- [ ] Нет emoji в ответах AI-агента
- [ ] Нет Unicode-графики в чате

## 14. Формат сигнатуры стека

- Размещение: правый нижний угол (только для корневых `README.md` и `CHANGELOG.md`)
- Формат: `Built with: <технологии проекта>` (конкретный стек определяется проектом, не стандартом)
- Пример (для Next.js проектов): `Built with: Next.js 16 + TypeScript + Tailwind CSS`
- Разрешены: латиница, кириллица, цифры, символы + и :
- Запрещены: emoji, Unicode-графика

**Примечание:** Стандарт определяет формат (структуру), а не конкретные технологии. Значение по умолчанию для проектов этого стека см. в `README_TEMPLATE.md`.
