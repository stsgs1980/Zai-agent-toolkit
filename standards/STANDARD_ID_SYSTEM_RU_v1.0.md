# Стандарт: Standard ID System v1.0 (RU)

> ID: STD-META-001
> Version: 1.0
> Level: **[W] Warning**
> Last Updated: 2025-01

Документ описывает систему идентификаторов для всех стандартов проекта. ID обеспечивает однозначную ссылку на стандарт при обсуждениях, в коде, в AI-промптах и документации.

---

## 1. Назначение

Система ID решает следующие задачи:

| Задача | Пример |
|--------|--------|
| Однозначная ссылка в обсуждениях | "Нарушение STD-FE-001 секция 2.1" |
| Ссылка в AI-промптах | "Следуй STD-GIT-001 для коммитов" |
| Связь между стандартами | No-Unicode Policy ссылается на STD-DOC-001 |
| Поиск и каталогизация | grep "STD-FE-" находит все упоминания Frontend Standard |

---

## 2. Формат ID

```
STD-<DOMAIN>-<NUMBER>
```

| Компонент | Описание |
|-----------|----------|
| `STD` | Префикс (Standard) |
| `<DOMAIN>` | Домен стандарта (2-4 буквы) |
| `<NUMBER>` | Порядковый номер в домене (3 цифры) |

**Примеры:**
- `STD-FE-001` — Frontend Development Standard
- `STD-GIT-001` — GitHub Operations Standard
- `STD-A11Y-001` — WCAG Accessibility Standard

## 5. Порядок назначения ID

### 5.1. Кто назначает

| Роль | Полномочия |
|------|------------|
| Tech Lead | Создание новых доменов, назначение ID |
| Любой участник | Предложение нового стандарта через Issue/PR |

### 5.2. Процесс

1. **Создать Issue** с предложением стандарта
2. Указать предполагаемый домен (или запросить новый)
3. Tech Lead назначает ID из реестра
4. После approval — стандарт добавляется с ID в заголовке

### 5.3. Правила нумерации

- Номера идут последовательно: 001, 002, 003...
- Пропуски не допускаются (нет STD-DOC-006 если нет STD-DOC-005)
- Удалённый стандарт: ID помечается как `[DEPRECATED]` в реестре, номер не переназначается

## 6. Формат заголовка стандарта

Все стандарты обязаны иметь заголовок в следующем формате:

```markdown
# Standard: <Название> v<Версия> (<Язык>)

> ID: STD-<DOMAIN>-<NUMBER>
> Version: <Версия>
> Level: **[<Уровень>] <Название уровня>**
> Last Updated: <Дата>
> Related: <Связанные стандарты по ID> (опционально)
```

**Пример:**

```markdown
# Standard: Frontend Development v1.1 (EN)

> ID: STD-FE-001
> Version: 1.1
> Level: **[C] Critical**
> Last Updated: 2023-10-27
```

## 8. Обновление реестра

### 8.1. Триггеры для обновления

| Событие | Действие |
|---------|----------|
| Новый стандарт | Добавить запись в реестр |
| Новая версия стандарта | Обновить колонку Version |
| Депрекация стандарта | Добавить `[DEPRECATED]` |
| Изменение уровня | Обновить колонку Level |

### 8.2. Место хранения реестра

Реестр ведётся в этом документе (STD-META-001), раздел 4.

### 8.3. Синхронизация

При изменении реестра:
1. Обновить этот документ
2. Убедиться, что ID в заголовке стандарта совпадает с реестром
3. Закоммитить изменения с сообщением `docs(meta): update standard registry`

## 9. Обратная совместимость

### 9.1. Изменение ID

ID **не изменяется** после назначения. Исключение — реорганизация доменов с явным депрекацией старого ID.

### 9.2. Удаление стандарта

При удалении стандарта:
1. ID помечается `[DEPRECATED]` в реестре
2. В файл стандарта добавляется предупреждение в заголовке
3. Номер не переназначается новому стандарту

## 10. FAQ

### Q: Зачем нужны ID если есть названия файлов?

A: Названия файлов могут меняться, файлы могут перемещаться. ID — постоянный идентификатор, который survives рефакторинг структуры.

### Q: Что делать если домен переполнен (999 стандартов)?

A: Разделить домен на поддомены (например, `FE-COMP` для компонентов, `FE-PERF` для производительности).

### Q: Можно ли использовать ID в CI/CD?

A: Да. Например, линтер может выводить: `Error: STD-FE-001 violation — component exceeds 150 lines`.
